Environment

  1. The ASC programming language is designed to be semantically independent from any real computing system. However, for the purpose of this standard, ASC programs are assumed to be interpreted using an abstract von Neumann computer architecture. This architecture is refered to as IISC (Intermediary Instruction Set Computer).

  2. IISC includes...

    • an abstract CPU architecture
    • an instruction set for the abstract CPU
    • a limited memory model scheme for facilitating structured program execution
    • a binary object format used for storing and distributing programs
    • a linking and loading system for executing programs stored as binary objects

  3. The ASC programming language is not required to be interpreted by or translated to IISC programs. Similarly, the IISC architecture may be used to interpret programs translated from any other language.

  4. IISC does not abstract interactions with the operating system. The abstract CPU architecture does not include many architecture specific behaviors that are common in real hardware. Functionality for, among other things, privilege levels, memory security, and ports are not considered. It is discouraged to extend an implementation of IISC with support for these functionalities.

  5. IISC allows for ASC program components to be stored in either a system-native or IISC-native encoding.

  6. IISC allows for ASC program components of any system-native or IISC-native format to be combined and executed as one cohesive program. IISC is designed to function transparently as part of the underlying operating system. It should be implemented to behave similarly to the native operating system when executing programs.
